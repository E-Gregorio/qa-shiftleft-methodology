GUÍA ESTRATEGIA DE PRUEBAS
QA Shift-Left Methodology · by E-Gregorio — Marco común alineado a ISTQB v4 / ISO/IEC/IEEE 29119
Esta guía define el marco común de pruebas para garantizar cobertura completa, gestión eficaz de riesgos y optimización de la automatización en todos los proyectos.

🎯 Objetivo y alcance

Aspecto Descripción
Objetivo principal Establecer un marco común de pruebas que garantice la calidad, eficiencia y predictibilidad en todos los proyectos
Alcance Aplicable a todos los desarrollos (web, mobile, APIs, microservicios) desde la fase de análisis hasta producción
Marco regulatorio Alineado con ISTQB v4, ISO/IEC/IEEE 29119 y ISO/IEC/IEEE 29119-3
Enfoque Shift-Left Testing con gestión de riesgos y automatización basada en criterios VCR

📊 Niveles de prueba

1️⃣ Pruebas Unitarias

Aspecto Detalle
Responsable Desarrollador (como parte de DoD)
Objetivo Verificar el comportamiento de componentes individuales aisladamente
Herramientas Jest, JUnit, NUnit, pytest, Mocha/Chai según tecnología
Cobertura mínima 80% de líneas de código (monitoreado en CI/CD)
Integración Ejecutadas automáticamente en cada commit y build

2️⃣ Pruebas de Integración

Aspecto Detalle
Responsable QA + Desarrollador
Objetivo Verificar la comunicación entre componentes, APIs y servicios externos
Tipos Big Bang, Incremental (top-down/bottom-up), Contract Testing
Herramientas Postman/Newman, RestAssured, Pact, TestContainers
Integración Ejecutadas en pipeline de integración continua

3️⃣ Pruebas de Sistema

Aspecto Detalle
Responsable Equipo QA
Objetivo Verificar el comportamiento end-to-end del sistema completo
Tipos Funcional, No funcional (performance, seguridad, usabilidad)
Herramientas Selenium, Playwright, Cypress, JMeter, OWASP ZAP
Ambiente Ambiente de testing que simula producción

4️⃣ Pruebas de Aceptación

Aspecto Detalle
Responsable Product Owner + Usuario Final
Objetivo Validar que el sistema cumple con los criterios de aceptación del negocio
Tipos UAT (User Acceptance Testing), BAT (Business Acceptance Testing)
Herramientas Cucumber/Gherkin, FitNesse, herramientas de gestión manual
Ambiente Ambiente de pre-producción con datos reales anonimizados

⚖️ Priorización basada en riesgo

Metodología VCR

Utilizamos el modelo VCR (Valor-Costo-Riesgo) para priorizar qué funcionalidades se automatizan:

  • V (Valor): Impacto de negocio (1-3)
  • C (Costo): Esfuerzo de ejecutar pruebas manuales repetitivamente (1-3)
  • R (Riesgo): Probabilidad × Impacto de falla (1-9)
Puntuación VCR Estrategia Acciones
VCR ≥ 9 Automatización obligatoria Se incluye en suite de regresión automática y deuda técnica
VCR 6-8 Automatización recomendada Evaluar en próximos sprints según capacidad del equipo
VCR < 6 Pruebas manuales Ejecutar en ciclos de testing manual, revisar en cada sprint

🔄 Estrategia de automatización

Tipo de prueba Nivel de automatización Frecuencia Herramientas
Pruebas de humo 100% automatizado Cada deploy Playwright TypeScript, Cypress
Regresión crítica 100% automatizado Nightly builds Playwright TypeScript, Selenium Grid
API Testing 95% automatizado Cada build Postman, locust
Pruebas E2E 70% automatizado Sprint end Playwright, Selenium
Pruebas exploratorias 0% automatizado On-demand Manual + herramientas de session recording

📈 Métricas y KPIs

🎯 Cobertura

  • Code Coverage: ≥80% (unitarias)
  • Functional Coverage: ≥95% (críticas)
  • Requirements Coverage: 100%

⚡ Eficiencia

  • Automation Rate: ≥70%
  • Test Execution Time: <30 min (regresión)
  • First Time Right: ≥85%

🔍 Calidad

  • Defect Density: ≤3 defectos/100 SP
  • Defect Leakage: ≤5%
  • Customer Satisfaction: ≥4.5/5

📊 Proceso

  • Test Case Pass Rate: ≥95%
  • Environment Uptime: ≥99%
  • MTTR (Mean Time to Repair): ≤2h

Developer Escribir pruebas unitarias, participar en estimación VCR, fix de defectos, code review
Product Owner Definir criterios de aceptación, priorizar funcionalidades, validar UAT, aprobar releases
Scrum Master Facilitar ceremonias, remover impedimentos, gestionar riesgos, seguimiento de métricas
